OBJETIVO DEL PROYECTO

Web app funcional para seguimiento de entrenamiento partiendo de un Excel existente.
La aplicacion permite:

- registro de usuarios con username propio
- login por email O por username
- persistencia real de datos en base de datos serverless
- catalogo de ejercicios
- registro de sesiones y series
- historial de entrenamiento
- dashboard de progreso
- ajustes de perfil: cambio de contrasena y borrado de registros
- futura exportacion a Excel

ENDPOINTS IMPLEMENTADOS (Next.js API Routes)

Publicos:
- POST /api/auth/register   body: { email, username, full_name, password }
- POST /api/auth/login      body: { identifier (email o username), password }
- GET  /api/exercises
- GET  /api/init?secret=X   inicializa tablas y seed (llamar una vez tras deploy)

Protegidos (JWT Bearer):
- GET  /api/sessions
- POST /api/sessions        body: { title, training_day, week_number, notes?, sets[] }
- GET  /api/dashboard
- GET  /api/measurements
- POST /api/measurements
- PUT  /api/profile/password   body: { current_password, new_password }
- DELETE /api/profile/records  borra todas las sesiones y series del usuario


VARIABLES DE ENTORNO REQUERIDAS (Vercel)

TURSO_DATABASE_URL=libsql://nombre-db.turso.io
TURSO_AUTH_TOKEN=token-de-turso
JWT_SECRET=cadena-larga-aleatoria-minimo-32-chars
INIT_SECRET=cualquier-secreto-para-proteger-init


PASOS PARA PONER EN PRODUCCION

1. Crear cuenta en turso.tech (gratuita)
2. Crear base de datos: turso db create gym-tracker
3. Obtener URL: turso db show gym-tracker --url
4. Obtener token: turso db tokens create gym-tracker
5. Anadir las 4 vars en Vercel -> Settings -> Environment Variables
6. Redeploy desde web-app/: vercel --prod
7. Llamar una sola vez: GET https://web-app-sigma-hazel.vercel.app/api/init?secret=<INIT_SECRET>
8. Verificar que devuelve { ok: true }


BASE DE DATOS (Turso SQLite serverless)

Tablas:
- users (id, email, username, full_name, password_hash, is_active, created_at)
- exercises (id, name, muscle_group, equipment)
- workout_sessions (id, user_id, title, training_day, week_number, notes, created_at)
- session_sets (id, session_id, exercise_id, position, weight_kg, reps, rir)
- body_measurements (id, user_id, measured_at, weight_kg, body_fat_percent, chest_cm, waist_cm, hip_cm, arm_cm, thigh_cm, notes)

Seed inicial: 12 ejercicios por grupos musculares (pecho, espalda, hombro, biceps, triceps, pierna, core).


COMO FUNCIONA EL FLUJO DE USUARIO

1. Entra en la home.
2. Crea cuenta en /register: nombre, username, email, contrasena.
3. El backend registra, hashea contrasena con bcryptjs, devuelve JWT.
4. Login en /login: puede usar email O username mas contrasena.
5. JWT guardado en localStorage (MVP, migrar a cookies HttpOnly en produccion).
6. Dashboard en /dashboard: resumen semana, sesiones recientes, tipo de semana (HIP/FUE/DEL).
7. Registro de sesion: dia, semana, ejercicios, kg/reps/RIR por serie.
8. Ajustes en /settings:
   - Cambiar contrasena (verifica la actual antes de actualizar)
   - Borrar todos los registros de entrenamiento (confirmacion inline antes de ejecutar)


CRITERIOS DE SEGURIDAD

- contrasenas hasheadas con bcryptjs (rounds: 10), nunca en texto plano
- JWT firmado con JWT_SECRET (minimo 32 chars recomendado)
- endpoints sensibles verifican JWT en cada request
- datos separados por user_id, imposible acceder a datos de otro usuario
- cascade delete en session_sets al borrar sesion
- /api/init protegido por INIT_SECRET, no accesible sin conocerlo

Limitaciones actuales:
- JWT en localStorage (no ideal para produccion, suficiente para MVP)
- no hay rate limiting en login (proteccion contra brute force pendiente)
- no hay verificacion de email real

ULTIMO PROBLEMA DETECTADO

La web app desplegada en Vercel mostraba "No se ha podido conectar con el servidor"
en todas las acciones (login, registro, dashboard).

Causa raiz:
El frontend Next.js estaba desplegado en Vercel pero el backend era FastAPI (Python)
que jamas se habia desplegado correctamente. La variable NEXT_PUBLIC_API_URL no estaba
configurada en Vercel, por lo que el frontend usaba el fallback http://localhost:8000
que obviamente no existe en internet. El api/vercel.json estaba vacio ({}) y FastAPI
nunca se ejecuto como funcion serverless.

COMO SE HA CORREGIDO

Se ha eliminado la dependencia del backend FastAPI para produccion.
Todo el backend se ha reescrito como Next.js API Routes (TypeScript) dentro del mismo
proyecto web-app. Ahora todo corre en el mismo proceso de Vercel:

- Se ha instalado @libsql/client para conectar con Turso (SQLite serverless)
- Se ha instalado jose para JWT edge-compatible
- Se ha instalado bcryptjs para hashing de contrasenas
- Se han creado 9 rutas API en app/api/
- Se ha creado src/lib/db.ts con cliente Turso lazy (no falla en build sin env vars)
- Se ha creado src/lib/migrate.ts para inicializacion de tablas y seed
- src/lib/config.ts ahora usa apiUrl = "" (rutas relativas, sin servidor externo)
- Se ha eliminado output: standalone de next.config.mjs (incompatible con Vercel serverless)
- Se han actualizado los formularios: username en registro, identifier en login
- Se ha anadido pagina /settings con cambio de contrasena y borrado de registros
- El build de Next.js pasa limpio con todos los routes marcados como dinamicos

Pendiente de completar: configurar las 4 variables de entorno en Vercel y llamar a /api/init.
