OBJETIVO DEL PROYECTO

Construir una web app funcional para el sistema Gym Tracker partiendo de un Excel existente. La aplicacion debe permitir:

- registro de usuarios
- login
- persistencia real de datos
- consulta de ejercicios
- registro de sesiones
- historial de entrenamiento
- dashboard de progreso
- futura exportacion a Excel

COMO FUNCIONA EL PRODUCTO AHORA

Flujo funcional actual del usuario:

1. El usuario entra en la home del frontend.
2. Puede abrir `/register` y crear una cuenta.
3. El formulario muestra desde el principio que pide nombre, email valido y contrasena segura.
4. Mientras escribe la contrasena, el frontend marca en vivo los requisitos cumplidos y pendientes.
5. El frontend llama al backend `POST /api/v1/auth/register`.
6. Si falla algun dato, el frontend muestra el error exacto junto al campo correspondiente.
7. La API crea el usuario y guarda la contraseña hasheada.
8. El frontend hace login automaticamente con `POST /api/v1/auth/login`.
9. Si falla el acceso, el frontend indica si el problema esta en email o contrasena.
10. El backend devuelve un token JWT.
11. El frontend guarda el token en localStorage para este MVP local.
12. El usuario entra en `/dashboard`.
13. El dashboard carga:
   - resumen del usuario
   - ejercicios disponibles
   - sesiones guardadas
14. El dashboard ofrece pestanas para:
   - vista general
   - entreno
   - medidas
   - graficas
15. En la pestana de entreno el usuario usa selectores para dia, semana, ejercicios y RIR.
16. El usuario puede anadir varias series dentro de una misma sesion.
17. La API guarda la sesion y sus series.
18. El dashboard se actualiza con el historial real.
19. En la pestana de medidas el usuario registra peso y perimetros corporales.
20. La API guarda esas medidas.
21. En la pestana de graficas el frontend representa tendencias de peso y volumen por sesion.

Flujo publico actual:

1. El frontend se sirve desde Vercel.
2. La API se ejecuta en Vercel Functions.
3. El usuario ya no necesita tener su PC encendido ni un `cmd` abierto para que la web responda.

DONDE SE GUARDA LA INFORMACION

- Base de datos local SQLite:
  `C:\claude\CARRERA PROFESIONAL\gym-tracker-system\api\data\gym_tracker.db`

- En despliegue Vercel, si no se configura `DATABASE_URL`, la API usa un SQLite temporal en `/tmp/gym_tracker.db`

ULTIMO PROBLEMA DETECTADO

La web resultaba demasiado basica y poco parecida al uso real del Excel: pocas opciones de ejercicio, demasiados campos manuales, falta de medidas corporales y ausencia de una zona de graficas. Ademas, el usuario detecto que un email como `prueba@prueba.prueba` podia registrarse y pidio aclarar si eso era deseado o no.

COMO SE HA CORREGIDO

- Se ha ampliado el seed de ejercicios hasta 26 movimientos repartidos por grupos musculares y equipamiento.
- El seed ahora anade ejercicios faltantes aunque la base ya exista, en lugar de quedarse bloqueado con el catalogo corto inicial.
- Se ha creado la tabla `body_measurements` con sus esquemas, repositorio, servicio y endpoints propios.
- Se ha rehecho el dashboard con pestanas para vista general, entreno, medidas y graficas.
- El registro de entrenamiento usa ahora selectores para dia, semana, ejercicio y RIR, y permite varias series por sesion.
- Se ha anadido una pestana de medidas corporales con peso, grasa, pecho, cintura, cadera, brazo y muslo.
- Se ha anadido una pestana de graficas con tendencias visuales de peso y volumen de entrenamiento.
- Se ha dejado explicado en el formulario de registro que en este MVP se valida el formato del email, no la existencia real del buzon.
- Se mantiene asi por ahora porque no hay pagos ni recuperacion de contrasena por email y endurecerlo de verdad exige verificacion de correo.
- Se ha recompilado el frontend.
- Se ha redeployado la API publica en `https://api-nine-ashen-92.vercel.app`.
- Se ha redeployado el frontend publico en `https://web-app-sigma-hazel.vercel.app`.
